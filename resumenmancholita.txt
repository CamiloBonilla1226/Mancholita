Público (sin login)

Listar productos activos con:

2) Modelo de datos que definimos (lo clave)

Category (tabla categories)

id

name

active

parent_id (self join) Esto permite estructura jerárquica:

Roots: Hombre(1), Mujer(2)

Children: Jeans/Camisas/Blusas apuntando a un parent.


Product (tabla products)

id, name, description, price, active

imageUrl

category_id FK a categories Relación: Product -> ManyToOne(Category).


👉 Gender no existe como tabla: lo inferimos con category.parent

Si la categoría del producto es “Jeans Mujer”, su parent es “Mujer”.


Orders y Order Items

Ya tienes tablas:

orders con datos del cliente, created_at, total, etc.

order_items con product_id, product_name, unit_price, quantity, line_total, y order_id.

3) Cómo funciona el flujo principal (producto)

Listado público /api/public/products

En vez de devolver Product entidad, devolvemos DTO optimizado ProductPublicDto.

Esto evita Lazy-loading, evita loops JSON, y entrega exactamente lo que el front necesita.

Query en el repositorio con select new ProductPublicDto(...):

trae categoryId/categoryName

trae genderId/genderName desde category.parent



Admin productos

El controller ya no habla con repositorios, habla con ProductService.

ProductService contiene reglas:

buscar categoría por id

validar que sea hoja (existsByParentId(category.getId()))

crear/actualizar

activar/desactivar con patch

4) Arquitectura que usamos (simple, limpia y práctica)

No hicimos “hexagonal full”, pero sí un layered architecture bien ordenado:

Capas

1. api: Controllers + DTOs + request/response


2. application: Services (reglas del negocio y orquestación)


3. domain: Entities JPA (Product, Category, Order, etc.)


4. infrastructure: Repositories (Spring Data JPA) + proyecciones + cosas de persistencia



Regla mental:

Controller -> Service -> Repository -> DB

Entidades no se exponen al público (se usan DTOs).

Validaciones de request con jakarta.validation.
